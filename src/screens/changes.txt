  - You also have duplicate-

  So the likely root issue is:

  - On update, the app changed
    download persistence/
    hydration behavior.
  - Old Room rows that
    previously did not matter,
    or were filtered out, are
    now being included by
    getActiveDownloads() /
    hydrateDownloadStore().
  - Those rows contain
    incomplete legacy metadata:
    wrong totalBytes, missing
    modelKey/quantization,
    maybe old “completed”
    semantics.
  - Hydration now trusts them
    enough to show them as
    completed-ish UI entries.

  Why this can suddenly appear
  after an update:

  - A migration changed which
    rows survive.
  - getActiveDownloads() now
    returns more rows than
    before, including completed
    or legacy rows.
  - JS hydration/filtering
    rules changed, so
    previously ignored rows are
    now surfaced.
  - Metadata fields added in
    newer versions are absent
    in old rows, causing
    fallback values like
    Unknown.

  Your key question about “file
  size check on every
  hydration”:

  - For 10 completed rows, a
    file stat check is usually
    not huge, probably tens to
    a few hundreds of
    milliseconds total
    depending on storage/
    device.
  - But doing it every launch/
    foreground is still the
    wrong shape of fix.
  - More importantly, it treats
    the symptom at the UI
    boundary instead of fixing
    the bad persisted rows.

  Simple solution that prevents
  this for users on update:

  1. Do a one-time cleanup on
     app update, not every
     hydration.
  2. Run it against the native
     Android download DB,
     because that is where the
     bad rows live.
  3. Delete rows that are
     clearly legacy-invalid,
     for example:
      - status = completed but
        totalBytes is
        implausibly small for
        known large models
      - missing critical
        metadata for completed
        rows
      - duplicate logical model
        where one row is
        obviously invalid and
        another valid
        downloaded model
        already exists
  4. After cleanup, hydration
     just reads clean data.
